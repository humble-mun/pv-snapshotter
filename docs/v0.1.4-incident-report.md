## 2. 当前现象（生产环境）

### 2.1 表现

- 部分 Pod 创建失败，报 `failed to create snapshot: missing parent "k8s.io/<seq>/sha256:<chainID>" bucket: not found`
- 不同节点上失败的镜像不同（每个节点 metadata 状态局部不一致）
- 已经在跑的容器**不受影响**（runc/内核持有 overlay mount 引用）
- `scale 0/1` 模式下，新 Pod 调度到失败节点会创建不出来；调度到健康节点能正常工作（命中"巧合是好的"组合）

### 2.2 节点级状态特征（用 `fix-meta` dry-run 可验证）

```
namespace="k8s.io"  pv-snapshotter bucket: N entries  -> 这些 entries 都是污染
namespace="moby"    pv-snapshotter bucket: not present
```

每个节点的 N 不同，污染的镜像名也不同。

### 2.3 关键约束

- **不能节点级别重置** containerd（节点上跑着 ceph 存储，重置 containerd 会让整个节点上所有容器重建，包括 ceph，灾难性）
- **不能去掉 webhook**（Pod 不再走 pv-snapshotter，重建后状态丢失，违背业务"关机/开机"语义）
- **不能停 pv-snapshotter daemon**（与 containerd 启动顺序耦合，停了之后 Pod 创建直接失败）

## 3. 根因分析

### 3.1 v0.1.4 的两个原始 bug（已在代码层面修复）

#### Bug 1: `rootPath` 改为专属子目录导致镜像层不可见

v0.1.3 用 `rootPath = /var/lib/containerd`，pv-snapshotter 的 overlay 实例和 containerd 主进程 overlayfs **共享 `snapshots/` 目录树**（虽然 `metadata.db` 是独立的）。这是 v0.1.3 工作的关键基础——所有镜像层在共享目录里，pv-snapshotter overlay 能 stat 到。

v0.1.4 改为专属子目录后，pv-snapshotter overlay 看不到 containerd 主 overlayfs unpack 的镜像层，每次 `Prepare()` 都报 `missing parent`。

**修复**：当前 HEAD 已回退到 `rootPath = /var/lib/containerd`。

#### Bug 2: `Cleanup()` 未转发导致磁盘永远不回收

`syncRemove=false`（默认）时，containerd `Remove()` 只删 BoltDB 元数据，物理目录推迟到 `Cleanup()` 调用时清理。`snapshotservice.FromSnapshotter` 通过 `snapshots.Cleaner` 类型断言分发 gRPC `Cleanup` RPC。pv-snapshotter 的 `snapshotter` struct 嵌入的是 `snapshots.Snapshotter` 接口（不是具体类型），断言永远失败返回 `ErrNotImplemented`。每个删除的容器都留下永久孤儿目录。

**修复**：当前 HEAD 引入 `cleanerSnapshotter`，启动时一次性做类型断言，`Cleanup()` 通过预检查的 `snapshots.Cleaner` 直接转发。

### 3.2 排查过程中引入的额外破坏（这是当前生产污染的真正来源）

#### 破坏 1: `unpack-image` 工具写入裸 chainID key

为修复 Bug 1 引入的工具，绕过 containerd metadata 层、直接连 pv-snapshotter socket 调用 `Prepare()`，参数 key 是裸 chainID（如 `sha256:abc...`）。但 containerd CRI 正常路径调 pv-snapshotter 时 key 是 metadata 层变换后的 bkey（`k8s.io/<seq>/sha256:abc...`）。两套 key 体系混在 pv-snapshotter metadata.db 里，互相干扰。

#### 破坏 2: 注入 `runtime_platforms` 改变 unpack 路径

让 kubelet 触发的 PullImage 走 pv-snapshotter unpack。但 unpack 失败时，containerd 主 metadata 已经分配了 sequence ID（写入 `v1/k8s.io/snapshots/pv-snapshotter/<chainID>/name = "k8s.io/<seq>/<chainID>"`），并在 image config blob 上写了 `gc.ref.snapshot.pv-snapshotter` label——但 pv-snapshotter 自己的 BoltDB 里没有对应记录。这就是所谓的 **bkey 不一致**。

#### 破坏 3: 注入 `transfer_unpack`（短暂引入又移除）

覆盖了 containerd 默认 `unpack_config`，导致 overlayfs 也失去 unpack 支持，所有镜像 pull 失败。已回滚。

#### 破坏 4: 多次手动操作 image 记录、清 label

为修复上述问题反复 `ctr images rm`、清 `gc.ref.snapshot.overlayfs/pv-snapshotter` label、删 `metadata.db`。每次操作都让 metadata 状态进一步偏离。

### 3.3 当前 stale 状态的本质

containerd 主 metadata BoltDB（`/var/lib/containerd/io.containerd.metadata.v1.bolt/meta.db`）里：

```
v1/k8s.io/snapshots/pv-snapshotter/<chainID>/name = "k8s.io/<seq>/<chainID>"
```

这种记录存在，但 pv-snapshotter 自己的 metadata.db 里没有 `k8s.io/<seq>/<chainID>` bucket。

新容器创建时：
1. CRI → containerd → metadata 层 `Prepare(key, parent=<chainID>)`
2. metadata 层查 `pbkt = bkt.Bucket([]byte(parent))` 找到 chainID bucket
3. `bparent = pbkt.Get(bucketKeyName)` 拿到 stale bkey `"k8s.io/<seq>/<chainID>"`
4. metadata 层调 `pv-snapshotter.Prepare(newBkey, bparent)`
5. pv-snapshotter 查自己的 metadata.db → 没有 `bparent` bucket → 报 `missing parent`

### 3.4 image config blob 上的 `gc.ref.snapshot.pv-snapshotter` label

调查（见 `bg_cc8b63d5` 输出）确认：
- 这个 label 由 `core/unpack/unpacker.go:640-643` 在 unpack 流程末尾**无条件写入**
- **没有后台 goroutine 自动重写**
- containerd 重启不会重写 label
- 但每次 unpack 流程（包括失败的 unpack）末尾都会写

也就是说：清 label 后，下一次 Pod 创建尝试触发 unpack → unpack 中途失败（`missing parent`）→ 但 label 写入仍然完成 → label 又出现。

## 4. 已经验证有效的修复（但代价不可接受）

**完全重置节点的 containerd metadata**：

```bash
systemctl stop kubelet
systemctl stop containerd
rm /var/lib/containerd/io.containerd.metadata.v1.bolt/meta.db
rm /var/lib/containerd/metadata.db
systemctl start containerd
systemctl start kubelet
```

**结果**：节点完全恢复，所有镜像被 kubelet 重新 pull，新建容器正常。
**代价**：节点上所有现有容器全部重建，包括 ceph 存储 Pod。生产环境**不可接受**。

## 5. 已经尝试过、确认无效的方向

### 5.1 删除 image 记录 → 让 kubelet 重新 pull

- `ctr images rm <image>` → kubelet PullImage → 期望走 `runtime_platforms` 重新 unpack
- **失败原因**：删 image 后 GC 把 layer blob 也删了。kubelet 触发的 pull 在 transfer service 层认为 content 已存在（manifest 还在），跳过下载，但 unpack 时找不到 layer blob。

### 5.2 清 image config blob 上的 `gc.ref.snapshot.pv-snapshotter` label

- `ctr content label <digest> "containerd.io/gc.ref.snapshot.pv-snapshotter="`
- **失败原因**：清除后，下一次 Pod 创建尝试触发 unpack，unpack 末尾无条件重写 label。

### 5.3 用 `fix-meta` 删除 `v1/k8s.io/snapshots/pv-snapshotter` bucket

- 工具：`docs/fix-meta/`（带 dry-run、备份、SHA-256 校验、原子事务）
- **失败原因**：删完后，下一次 Pod 创建尝试触发 unpack，unpack 中 metadata 层发现 chainID 不在主 BoltDB 里，重新分配 sequence + 创建 bucket（写入 `name = "k8s.io/<seq>/<chainID>"`），但实际 pv-snapshotter 的 unpack 失败（因为 layer blob 也已被删）→ 主 metadata 留下新的 stale 引用，循环。

### 5.4 强制重新 pull 镜像（绕过 ctr 的 cache 检查）

- 节点内 `ctr images pull` 不带 `--local` 走 transfer service（containerd 主进程处理 TLS）
- 节点内 `ctr images pull --local -k` 走 ctr 自己的 HTTP 客户端
- nsenter 进 host network namespace 后用 `--local -k` 是唯一稳定能 pull 的方式
- **部分有效**：能补回 layer blob，但和 5.1 / 5.3 的副作用一样，无法打破 stale 引用循环。

### 5.5 注入 `transfer_unpack`

- 在 `[[plugins."io.containerd.transfer.v1.local".unpack_config]]` 注入 pv-snapshotter
- **失败原因**：默认配置只包含 overlayfs 和 erofs，注入后**完全替换**默认列表，导致 overlayfs 失去 unpack 支持，pv-snapshotter DaemonSet 自身镜像也 pull 不下来。
- 后续修正为同时注入 overlayfs 条目，但仍无法修复 stale 引用问题。
- **已从代码中移除**。

## 7. 关键未决问题

### 7.1 v0.1.3 时代的 unpack 实际是怎么工作的？

代码看 `pv-snapshotter` 的 metadata.db 是独立的（`/var/lib/containerd/metadata.db`），containerd 主 overlayfs metadata.db 是独立的（`/var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/metadata.db`）。两者不共享。

那 v0.1.3 部署后：
- kubelet PullImage（不带 `--snapshotter pv-snapshotter`）→ containerd 主 overlayfs unpack → 镜像层在 overlayfs metadata.db
- 容器创建时 CRI 通过 RuntimeClass 走 pv-snapshotter → pv-snapshotter `Prepare(parent=<chainID>)`

按理说 pv-snapshotter 找不到 parent。但 v0.1.3 据用户陈述工作正常。

**待确认**：v0.1.3 时代节点上 `/etc/containerd/config.toml` 是否把全局 `snapshotter` 设成了 `pv-snapshotter`（不是 RuntimeClass 级别）？如果是，那一切镜像 unpack 都走 pv-snapshotter，bkey 一致。代价是 pv-snapshotter 单点故障会拖死所有 Pod。

请获取一台未被破坏的节点（如果还有的话）的 containerd config 全文，或者从 git 历史里翻 Helm chart 的 ConfigMap 模板，确认 v0.1.3 时代真实的 containerd 配置。

### 7.2 是否存在精准修复 metadata 的算法？

理论上可行的算法：

```
对每个污染节点:
  1. 列出 containerd 主 metadata 里 v1/k8s.io/snapshots/pv-snapshotter/<chainID>
     的所有 chainID
  2. 对每个 chainID:
     - 检查 pv-snapshotter 自己的 metadata.db 里是否有对应 bucket
     - 如果没有: 这是 stale 引用
     - 但**不能**直接删 stale 引用：删了之后下次 unpack 又会创建新的 stale
  3. 关键：阻止下次 unpack 重新写 stale
     - 只有让 unpack 真正成功才能打破循环
     - 让 unpack 成功需要 layer blob 存在 + parent chainID 链完整 + pv-snapshotter
       自己 metadata.db 里有正确 bucket
```

链条最薄弱的环节是：**pv-snapshotter 自己 metadata.db 里的 bucket 怎么和 containerd 主 metadata 同步**。当前 unpack 流程中，主 metadata 和 pv-snapshotter metadata 是两次独立写入，中间任何失败都会导致不一致。

## 8. 后续可以尝试的方向

### 8.1 最稳健（推荐）：分批节点级重置 + 业务感知

**前提**：和 ceph、业务方协调，节点逐个进入维护窗口。

```bash
# 单节点流程（在维护窗口内）
1. cordon 节点
2. drain 节点（migrate ceph osd 到其他节点；移走业务 Pod）
3. 完全重置 containerd metadata
4. uncordon 节点
5. 等节点上的 Pod 全部 ready 再做下一台
```

**优点**：100% 可恢复，无残留。
**缺点**：需要业务/ceph 配合，时间长。

### 8.2 验证 v0.1.3 实际配置（关键）

按 7.1 节获取 v0.1.3 真实 containerd 配置。如果是 **全局 `snapshotter = "pv-snapshotter"`**，那么：

```bash
# 在污染节点上
1. 确认 v0.1.3 配置
2. 把当前 config.toml 改回 v0.1.3 的全局配置
3. 删除 pv-snapshotter 自己的 metadata.db（清空 pv-snapshotter 状态）
4. 删除 containerd 主 metadata 里 pv-snapshotter 的所有 stale 引用
5. 让 kubelet 重新 PullImage（这次走全局 pv-snapshotter）
```

这是最有希望的非重置方向，但需要先确认 v0.1.3 的真实配置。

### 8.3 修改 pv-snapshotter 代码：在 Prepare 收到不存在的 parent 时，自动创建空 parent bucket

```go
// 伪代码
func (sn snapshotter) Prepare(ctx, key, parent string, opts...):
    if parent contains "k8s.io/<seq>/sha256:" pattern:
        if !pvSnapshotterMetadataDB.Has(parent):
            // 自动创建一个空的 parent bucket，指向已 unpack 的 overlayfs layer
            createEmptyParentBucket(parent)
    return underlying.Prepare(ctx, key, parent, opts...)
```

**风险**：理论可行但需要深入理解 BoltDB schema，破坏性改动；需要在测试环境充分验证。

### 8.4 用 `fix-meta` 工具增强：原子操作清 stale + 阻止重写

工具增强：
1. 删 `v1/k8s.io/snapshots/pv-snapshotter` bucket
2. 清所有 image config blob 上的 `gc.ref.snapshot.pv-snapshotter` label
3. **同时**清 `gc.ref.snapshot.overlayfs` label（让 image 看起来完全没 unpack）
4. 立即停 containerd
5. 删除 pv-snapshotter 自己的 metadata.db
6. 启动 containerd

观察这种"全清"是否能让 unpack 真正从头开始。**但 7.2 已经分析过，这种做法的失败模式是 unpack 中途失败再次留下 stale**。除非配合 8.3 的代码改动让 unpack 不再失败。
