## 0. TL;DR(一页结论)

1. **proxy + RuntimeClass 单位的架构本身是健全的,不需要 auto-unpacker、runtime_platforms、也不需要全局 snapshotter。**
   关键机制:**containerd 的 CRI 会在 `CreateContainer` 时,把镜像按需 unpack 进该容器所用的 snapshotter(即 pv-snapshotter)**。这一条在 dev(containerd 2.3.1)和生产版本(2.2.1)上都实测成立。
2. **v0.1.4 生产灾难的根因是 metadata / image-record 的"污染",不是设计缺陷。** 代码 HEAD(v0.1.3 + Cleanup 转发修复)是正确的。
3. **生产其实比事故报告描述的健康得多。** 实测两台生产节点(2.2.1)均健康:stale bucket=0、真实 runc-pv workload 零干预启动、未观测到活跃死锁。
4. **"双份数据"真实存在但作用域有限**(只针对 runc-pv pod 用到的镜像),用户已接受;消除它是可选优化(见"案C"),且能顺带砍掉大镜像启动时第二次 unpack 的耗时。
5. **不要用 `ctr images check` 的 "incomplete" 作为"镜像损坏/死锁"的判据**——它会把健康可运行的镜像也标成 incomplete(实测铁证见 §7.5)。基于它的自动删除方案已被否决并回退。
6. **下一步建议:金丝雀复原**——在单节点重新启用 pv-snapshotter 入口、跑真实 workload 观察;仅对"确实在用且 amd64 层确实缺失"的个别镜像做定点 `ctr images rm` 重拉。不需要大规模 de-pollution。

## 1. 背景:会话开始时的现状

### 1.1 pv-snapshotter 是什么

containerd 的一个 **proxy snapshotter(gRPC 插件)**,以 DaemonSet 部署在每个节点。核心功能:
当 pod 带有特定注解(由 mutating webhook 注入)时,在 overlay 挂载阶段把容器的**可写层 upperdir/workdir 重定向到一个 PV(CSI 卷)**,从而让容器的整个可写层落在 PV 上,具备"关机/开机"语义(pod 删除重建后可写层数据仍在 PV 上)。不带注解时透明 pass-through 到原生 overlay。

### 1.2 接入方式(关键事实)

- 通过 `[proxy_plugins.pv-snapshotter] type="snapshot"` 注册;只有 `runc-pv` 这个 RuntimeClass 的 `snapshotter = "pv-snapshotter"`。**默认 `runc` 仍走内置 overlayfs**(非全局)。
- daemon 内部用 `overlay.NewSnapshotter(rootPath=/var/lib/containerd)` **实例化了自己的 overlay**,有**独立的** `metadata.db`(`/var/lib/containerd/metadata.db`)和 `snapshots/`(`/var/lib/containerd/snapshots/`)。
- 主 containerd 的 overlayfs 在 `/var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/` 下(目录与 metadata 都独立)。
- 重定向逻辑只在 `Mounts()` 阶段:读 pod 注解 → 渲染 upperdir 路径 → 替换 overlay 挂载选项里的 `upperdir=`/`workdir=`。
- 相关代码:`pkg/containerd/snapshotter/{service.go,overlay.go,resolver.go,mount.go}`、`pkg/containerd/config/patcher.go`。

### 1.3 会话开始时用户描述的问题

- 生产环境(灾难发生的集群)经 `docs/recover-v0.1.4` 方案恢复后**仍未"彻底"修复**:
  - 节点上**已存在**某镜像、pod 刚好用它 → 能正常启动;
  - 节点上**没有**该镜像 → pod 起不来,且**未触发 kubelet 自动拉取**;手动 `ctr image pull` + `ctr image pull --snapshotter=pv-snapshotter` 后才能起;
  - `ctr image pull --snapshotter=pv-snapshotter` 不下载数据(已被前一条下载完),但会**卡很久**才显示 `unpacking ... done`;
  - 数据**双份**(overlayfs 一份、pv-snapshotter 一份),用户表示双份可接受。
- 因为生产始终没解决,已**摘除 pv-snapshotter 入口**(所有新建 pod 都用非 -pv 的 runtime class),功能当前在生产是关闭的。
- 当时挂着一批未提交改动(auto-unpacker,即 `pkg/containerd/snapshotter/imageunpacker.go` + service 接线),用户判断"基本无实质作用"。

### 1.4 预期结果

- 追求"最优解"的合理方案,并在**开发测试集群**完成验证(此前都是直接在生产节点上试)。
- 先做技术选型,再开发。明确不希望引入新的单点故障。

## 2. 系统架构与"双份数据"的本质

- 镜像层(只读 lowerdir)会被 unpack **两次**:kubelet 正常 PullImage 时 unpack 进 overlayfs(CRI 默认 snapshotter),容器创建(runc-pv)时再 unpack 进 pv-snapshotter。两套独立目录 → 双份。
- 实测确认(dev,见 §7.2):同一镜像 config blob 上**同时**挂着
  `containerd.io/gc.ref.snapshot.overlayfs=<chainID>` 与
  `containerd.io/gc.ref.snapshot.pv-snapshotter=<chainID>`,且**两者 chainID 完全相同**。
- 但作用域有限:`ctr snapshot --snapshotter pv-snapshotter ls` 只有 9 条(466M),而 overlayfs 有 252 条(3.5G)——**pv-snapshotter 不是 overlayfs 的全量镜像**,只装 runc-pv pod 实际用到的镜像。

---

## 3. 核心机制发现:CRI 在 CreateContainer 时按需 unpack

这是整次会话**最关键的发现**,解开了"v0.1.3 为何能工作"的谜题。

- 证据 1(dev,containerd 2.3.1):用节点上**不存在**的全新镜像创建 runc-pv pod,**零人工干预**直接起来。日志里出现
  `snapshotter does not support rebase capability, unpacking will be sequential`,
  紧跟在 `CreateContainer` 之后。
- 证据 2:该日志来自 vendored `core/unpack/unpacker.go:752`,即 **unpacker 在容器创建过程中运行了**。
- 证据 3(生产 2.2.1,gpu030):4.8GB 的 jupyterlab 镜像,kubelet 先 PullImage(58s),随后 `CreateContainer` **耗时约 53s 返回**——这 53s 就是把镜像按需 unpack 进 pv-snapshotter 的时间(顺序 unpack 大镜像)——然后 StartContainer 成功。

**含义:**
- proxy + RuntimeClass 模型天然可用,**不需要** auto-unpacker / runtime_platforms / 全局化。
- v0.1.3 能工作不是"巧合",而是这个 CRI 按需 unpack 机制(不是事故报告 §7.1 猜的"全局 snapshotter")。
- 事故报告 §3.1 的"v0.1.3 共享 snapshots/ 目录"说法**技术上是错的**:`missing parent` 是 metadata 层 bucket 有无的问题,与物理目录是否共享无关。
- 用户最初的"卡很久"= 第二次 unpack(进 pv)的固有成本,不是坏。

## 4. 技术选型讨论(及结论)

| 方案 | 描述 | 结论 |
|------|------|------|
| 案1:client 委托原生 overlayfs | pv-snapshotter 通过 containerd client 调 `SnapshotService("overlayfs")` | **否决**。proxy 已被 metadata 层包了一次,再调会**二次穿过 metadata 层**,产生两套 bkey、Mounts 时无法关联(当年就卡在这)。本质死路。 |
| 案2:in-tree 改 containerd 内置 snapshotter | fork containerd,编译进内置 snapshotter | 备选。能彻底消除独立 daemon 的鸡蛋问题、可设计成复用 overlay 存储消双份,但**维护 containerd fork 成本高**。仅当 proxy 模型确证不够时再考虑。 |
| 案3-全局化:把 pv-snapshotter 设为全局 default snapshotter | 所有镜像都经 pv 单一 snapshotter | **否决**(用户)。会让**所有 pod**(含 ceph、系统组件)都依赖 pv daemon,daemon 挂则整节点新容器全废 —— SPOF 比现状更严重。 |
| 案3-代理(维持现状) | proxy + RuntimeClass,靠 CRI 按需 unpack | **采纳**。§3 已证明它本就可用。SPOF 有作用域(只影响 runc-pv pod,非 -pv pod 走 overlayfs 不依赖 daemon)。 |
| 案C:薄共享 | pv 不自存只读层,按 **chainID** 复用 overlayfs 的层目录,只把 upper 落 PV | **可选优化(二期)**。消双份 + 砍掉大镜像第二次 unpack 的耗时。按 chainID(内容寻址、跨 snapshotter 同一键)关联,与案1 按 bkey 关联本质不同,技术上成立但复杂度高,需充分验证。 |

未提交的 `imageunpacker.go` + service 接线,思路是"watch 镜像事件,额外 `image.Unpack(pv-snapshotter)`"。§3 证明 CRI 已在容器创建时做了这件事,**P1 冗余 → 已破弃**(`git checkout` 回退 service/values/daemonset,删除 imageunpacker.go;保留无关的 `.gitignore` 改动)。

### 5.2 dev 全新镜像验证(containerd 2.3.1 / k8s 1.36,节点 chief)
`ubuntu:resolute-20260421`(全新)→ PullImage 17.5s → CreateContainer 触发按需 unpack → 启动成功。证明干净节点零干预可用。

### 5.3 双份数据确认
见 §2。`gc.ref.snapshot.{overlayfs,pv-snapshotter}` 同指一个 chainID;快照计数 9 vs 252。

### 5.4 recover-v0.1.4 再诊断(为什么没"彻底"修复)
`recover.sh` 只做:fix-meta 删主 metadata 里的 stale `v1/<ns>/snapshots/pv-snapshotter` bucket + 重灌 **pause** 一个镜像,其余靠 CRI 按需 unpack。缺口:
- **缺口1**:没补被 GC 回收的 layer blob;
- **缺口2(死锁核心)**:image 记录(manifest)还在 → kubelet 认为镜像已存在 → 跳过 PullImage → blob 永不补回 → 按需 unpack 失败(事故报告 5.1);
- **缺口3**:失败的 unpack 末尾仍回写 `gc.ref.snapshot.pv-snapshotter` label,可能再生 stale。

### 5.5 增强 recovery 尝试 → 在 2.2.1 上暴露版本差
曾给 recover.sh 加"删 blob 缺失的 image record"步(用 `ctr images check --platform ... | grep incomplete | ctr images rm`)。在 gpu030(2.2.1)dry-run 跑出来:
- **`ctr images check` 在 2.2.1 没有 `--platform` 标志**(报 `flag provided but not defined: -platform`)。只有 `-q`、`--snapshotter`。这正是"dev 2.3.1 能过、生产 2.2.1 会死"的版本差陷阱。
- gpu030 本身健全(stale=0、pause 正常 unpack),验证不了修复的"阳性路径"。

### 5.6 `ctr images check` 实测:incomplete ≠ 损坏(否决自动删除方案)
在 gpu030、gpu021 跑 `ctr -n k8s.io images check`(只读、不停 containerd):
- **铁证**:`harbor.smoothcloud.com.cn/system/pv-snapshotter-recover:v2` 自身显示 `incomplete (5/6) 1.9 MiB/30.2 MiB`,**但它刚在 gpu030 正常跑完了 recovery**。一个能运行的镜像被标 incomplete。
- 多数 incomplete 其实健康:`csi-*` 是 `incomplete (5/14)` 但 **94–98% 数据在**(缺的是 attestation/SBOM/其他架构层等不影响 amd64 运行的内容)。
- **结论**:`ctr images check` 的 incomplete 把多架构 index 的其他平台层、attestation 等也算 missing,**不能**当作"amd64 跑不起来/死锁"的判据。基于它"一刀切删 incomplete"会误删健康/在用镜像、甚至删掉 recovery 镜像本身。**该自动修复步已回退,daemonset-enhanced.yaml 已删除。**

### 5.7 生产健康度抽样(gpu030 / gpu021,containerd 2.2.1 / k8s 1.34)
- 两台都健康:workload 正常,gpu030 的 runc-pv 测试(jupyterlab)零干预起来。
- stale pv-snapshotter bucket = 0(fix-meta 无事可做)。
- 确有 amd64 层真缺的镜像(`cephcsi:v3.16.2` 296/824MiB、`hami:v2.7.0`),但**都不是在用版本**(在用的 `cephcsi:v3.16.1`、`hami:...-smoothcloud` complete 且在跑)→ 潜在、当前无害。

## 6. 阶段结论

1. 架构定性:proxy + RuntimeClass **正确且够用**;不需要 auto-unpacker/runtime_platforms/全局化。
2. 根因定性:生产灾难 = **污染**,非设计缺陷;代码 HEAD 正确。
3. 生产现状:抽样的两台节点**健康**,无活跃死锁;污染要么自愈、要么远没有"几百台全污染"那么严重。
4. 双份数据:真实但有作用域,用户接受;消除属可选优化(案C),附带启动提速。
5. 工具教训:**不要把 `ctr images check` 的 incomplete 当损坏判据**;任何自动修复必须按 **node 平台**解析 manifest、只校验其 config+layers(`ctr images check` 表达不了 → 需 platform-scoped 的工具,例如小 Go 工具)。

## 8.5 新故障模式:大镜像第二次 unpack 被 `context canceled`(2026-05-29 新增,已定位)

### 现象
某问题节点上 runc-pv pod 起不来,报:
```
failed to create containerd container: error unpacking image: apply layer error for
"harbor.smoothcloud.com.cn/sc13533561361/accelerate:5-20260501200630":
failed to extract layer sha256:13ffd99907aff5...: context canceled
```
这与 v0.1.4 主事故(元数据污染 → 镜像缺失 → 快速 fail)是**相反的失败模式**:不是"找不到东西快速失败",而是"unpack 正在进行但太慢 → 中途被取消"。

### 根因(已用 containerd v2.2.1 源码逐跳验证,生产同版本)
这条报错走的是 **CreateContainer 时的第二次 unpack**,不是 PullImage:
- `internal/cri/opts/container.go:47` —— `WithNewSnapshot` 在快照不存在(`errdefs.IsNotFound`)时调 `i.Unpack(ctx, c.Snapshotter)`,这就是"把镜像第二次 unpack 进 pv-snapshotter"那一步。
- `client/image.go:346-350` —— 该 `Unpack()` 顺序遍历各层调 `rootfs.ApplyLayerWithOpts(ctx, ...)`,失败时包成 `apply layer error for %q: %w`,**正是用户看到的报错串**。
- 该 `ctx` 直接来自 **CreateContainer 的 gRPC handler**,`container_create.go` 里**没有任何 `context.WithTimeout/WithDeadline`**——CreateContainer 不自带超时,沿用 kubelet 调用时挂的 deadline。
- 因 pv-snapshotter **不宣告 `rebase` 能力**(`core/unpack/unpacker.go:732-741`),整条 unpack 走**顺序模式**(§3/§5.7 已确认):**所有层共用同一个 ctx**(`unpacker.go:170-180,302-310`),一个 deadline 罩住全部层的串行解压。
- proxy snapshotter(`core/snapshots/proxy/proxy.go:84,101`)的 `Prepare()/Mounts()` 把 ctx 原样透传给 gRPC 调用,**没有 per-call 超时**——ctx 一旦 Done,正在进行的 `Mounts()`/`Apply()` 立即被取消。

### 回答用户的核心问题
**「pv-snapshotter grpc unpack 时间过长,containerd 侧是否有 timeout 主动关闭连接?」**
- containerd 的 CreateContainer 路径**本身不设超时**,proxy snapshotter 也**没有 per-call 超时**。
- 真正的 deadline 来自 **kubelet 的 `RuntimeRequestTimeout`(默认 2m)**(`k8s.io/kubelet/config/v1beta1/types.go:401`):它给整个 CreateContainer RPC 挂 deadline,2 分钟到点 → ctx cancel → 顺序 unpack 中途的 `Apply layer` 抛 `context canceled`。
- 注意:`image_pull_progress_timeout`(默认 5m,`internal/cri/config/config.go`)**不管这条路径**——它只监控 **PullImage 的网络字节进度**,而本次镜像 kubelet 已 pull 成功,失败的是 CreateContainer 时的第二次本地 unpack,二者上下文不同。**别去调 `image_pull_progress_timeout`,无效。**

### 为什么大镜像才中招
- 小镜像第二次 unpack 几秒内完成,2m 够用。
- `accelerate` 这种大镜像,顺序解压 amd64 各层总耗时可能逼近/超过 2m;一旦超过 kubelet 的 `RuntimeRequestTimeout`,健康的"慢但正常"的第二次 unpack 直接变成**硬失败**。这与 §3 jupyterlab ~53s 的观测一致——只是 `accelerate` 更大,越过了 2m 线。

### 缓解(按优先级)
1. **首选——调大 kubelet `runtimeRequestTimeout`**(如 5m–10m):
   ```yaml
   # kubelet config
   runtimeRequestTimeout: 10m
   ```
   这是唯一罩住 CreateContainer 第二次 unpack 的 deadline,直接对症。
2. **治本(二期)——案C(§8.3)**:按 chainID 复用 overlayfs 只读层,**彻底消掉第二次 unpack**,大镜像不再有这段串行解压,根除超时窗口。
3. ❌ **不要**调 `image_pull_progress_timeout`(走的不是这条路径,无效)。
4. ❌ pv-snapshotter **无法**靠"宣告 rebase 能力"转并行——它不支持 rebase,谎报会破坏 unpack 正确性。
5. (可选)pv-snapshotter 侧把 `Mounts()` 的 annotation 解析加缓存、`ensureUpperdirReady` 预校验,压低每层的 snapshotter 调用耗时,给顺序 unpack 争取时间——是次要优化,不解决根本。

### 待验证
- 该问题节点的 `accelerate` 镜像第二次 unpack 实测耗时(看 containerd 日志 CreateContainer 起止时间戳),确认是否确实跨过 2m。
- 调大 `runtimeRequestTimeout` 后该 pod 能否正常 Running(直接验证 deadline 假设)。

## 9. 已知陷阱 / 不要再踩

- ❌ 别把 pv-snapshotter 设为**全局** snapshotter(全节点 SPOF)。
- ❌ 别再加 auto-unpacker(CRI 已在容器创建时按需 unpack,冗余)。
- ❌ 别用 `ctr images check` 的 incomplete 当损坏判据,更别据此批量 `ctr images rm`(会误删健康/在用/recovery 镜像)。
- ❌ 别用案1(client 委托 overlayfs,按 bkey 关联)——二次穿 metadata 层,死路。
- ❌ 恢复时别动 `/var/lib/containerd/snapshots`(pv 的只读层)和 pv 的 `metadata.db`,会破坏运行中容器的 overlay 挂载。
- ⚠️ 版本差:生产是 **containerd 2.2.1 / k8s 1.34**,dev 是 2.3.1 / 1.36。任何工具/命令(如 `ctr images check --platform`)都要在 **2.2.1** 上验证。
- ❌ 大镜像 `context canceled`(§8.5)别去调 `image_pull_progress_timeout`——失败在 CreateContainer 第二次 unpack,不在 PullImage,该参数管不着。正确旋钮是 kubelet `runtimeRequestTimeout`。
- ❌ 别想靠让 pv-snapshotter 谎报 `rebase` 能力来转并行 unpack——不支持 rebase,谎报会破坏正确性。
